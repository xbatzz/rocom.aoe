# 个人版改造路线图

目标：将 fork 自 `aoe-top/rocom.aoe.top` 的项目，逐步改造成个人自用的“洛克王国世界战斗资料助手”。核心方向是属性克制、图鉴、技能与后续 PVP 对位辅助；不把每日任务、每周任务、打卡提醒作为核心方向。

## 基本原则

- `public/data` 和 `scripts` 视为上游基础数据层，个人功能不要直接修改。
- 基础数据更新优先使用 `yarn sync:pet-data`，不要手工大量改 JSON。
- 个人改造优先通过新增文件、新增组件、新增 `features` 目录实现，降低同步上游时的冲突。
- 暂时不做收藏、备注、每日/每周任务、打卡提醒等个人数据功能。
- 设计目标不是功能堆叠，而是更清晰、更有审美、更适合网页、Windows PC、Mac、iPhone、iPad 多端使用。

## 建议目录

```text
src/features/my-home
src/features/battle-query
src/features/pvp
src/features/design-system
src/custom
public/my-data
```

`src/features/my-home` 用于个人首页入口；`battle-query` 用于属性、图鉴、技能查询体验；`pvp` 用于后续队伍导入导出和对位助手；`design-system` 用于统一视觉规范；`src/custom` 或 `public/my-data` 仅在确实需要个人数据时使用。

## Phase 1：快速查询首页与视觉方向

当前状态：已完成首页第一版入口改造，核心入口聚焦属性克制、图鉴搜索、技能搜索，并保留 PVP Coming Soon。

### 目标

把首页改造成高频战斗资料入口，优先服务属性克制、图鉴搜索、技能搜索，并建立个人版视觉方向。

### 要做的功能

- 首页优先展示“属性克制”“图鉴搜索”“技能搜索”三个快速入口。
- PVP 工具先作为 `Coming Soon` 入口，避免过早实现复杂逻辑。
- “更多工具”区域保留原项目其他功能：配队、配种、孵蛋、星图、表格、道具、图鉴进度等。
- 不删除原有功能，只调整个人首页的信息层级。
- 优化首页多端布局，兼顾桌面、iPad、iPhone。

### 暂不做的内容

- 不做 PVP 真实计算。
- 不做个人收藏、备注。
- 不做每日任务、每周任务、打卡提醒。
- 不重构 `public/data` 和 `scripts`。

### 验收标准

- 首页首屏能直接进入属性克制、图鉴、技能查询。
- PVP 入口可见但明确标注为规划中。
- 原有工具仍可通过“更多工具”访问。
- 手机、平板、桌面布局不拥挤、不遮挡。

### 对上游同步风险的评估

风险中等。主要会改首页和可能新增个人首页组件。建议把新增 UI 放到 `src/features/my-home`，尽量减少对原页面大面积改动。

## Phase 2：属性、图鉴、技能查询体验优化

### 目标

优化最常用的三类查询：属性克制、图鉴、技能，让战斗中查资料更快。

### 要做的功能

- 优化属性克制查询体验，强化进攻、防守、倍率的快速识别。
- 优化图鉴搜索入口，支持更明显的名称、编号、属性搜索路径。
- 优化技能搜索入口，可基于现有 `moves.json` 或 `PetSkillIndex.json` 规划独立技能查询体验。
- 适配手机、iPad、桌面三类常用屏幕。
- 对高频结果使用更清晰的属性标签、技能类型标签和卡片层级。

### 暂不做的内容

- 不做个人数据存储。
- 不做技能收藏、宠物备注。
- 不做复杂伤害公式。
- 不手工改 `public/data` 中技能、图鉴、属性 JSON。

### 验收标准

- 属性关系页能更快读出克制、抵抗、被克制关系。
- 图鉴搜索入口比当前更突出。
- 技能查询有清晰入口和结果展示方案。
- 多端下输入、筛选、结果卡片都易操作。

### 对上游同步风险的评估

风险中等偏高。属性、图鉴、表格等页面是上游也可能修改的核心页面。建议优先新增 `src/features/battle-query` 组件，再由页面轻量接入。

## Phase 3：设计系统统一

### 目标

统一个人版视觉语言，让项目更像战斗资料助手，而不是多个工具页面的简单集合。

### 要做的功能

- 梳理颜色、卡片、按钮、属性标签、技能标签、状态徽章。
- 统一移动端间距、列表密度、卡片圆角、标题层级。
- 规划属性色、技能类别色、危险/优势/中性关系色。
- 规划后续 `docs/DESIGN_SYSTEM.md`，但本阶段先在路线图中定义方向。
- 保持设计感，同时不影响查询效率。

### 暂不做的内容

- 不引入新 UI 依赖。
- 不做全站一次性大重构。
- 不为了视觉效果牺牲移动端可读性。

### 验收标准

- 核心页面的按钮、卡片、标签风格一致。
- 属性和技能相关信息能通过颜色和层级快速识别。
- iPhone、iPad、桌面都有合理间距和信息密度。

### 对上游同步风险的评估

风险中等。样式和组件容易和上游 UI 更新冲突。建议新增 `src/features/design-system` 或少量 CSS 变量/工具类，逐步替换，不做大面积同时修改。

## Phase 4：PVP 队伍数据导入/导出

### 目标

建立 PVP 工具的数据基础：支持导入/导出我的队伍 JSON，但不写入上游基础数据区。

### 要做的功能

- 设计队伍数据结构，引用宠物 ID、技能 ID、性格、定位等稳定字段。
- 支持导入我的队伍 JSON。
- 支持导出当前队伍 JSON。
- 队伍数据放在 `localStorage`、`src/custom` 或 `public/my-data`，不写入 `public/data`。
- 为后续对位助手预留我方队伍、技能、属性、伤害观测字段。

### 暂不做的内容

- 不做复杂伤害公式。
- 不做联网同步。
- 不把个人队伍混入 `Pets.json`、`pets/*.json`、`moves.json`。
- 不做每日/每周队伍提醒。

### 验收标准

- 可以导入合法队伍 JSON 并显示队伍。
- 可以导出当前队伍 JSON。
- 数据结构文档清晰，能说明字段含义。
- 删除或替换个人队伍不会影响基础游戏数据。

### 对上游同步风险的评估

风险低到中等。只要放在 `src/features/pvp` 和个人数据区，基本不会影响上游同步。需要注意不要改动现有 `team.vue` 太多，优先独立实现 PVP 队伍模块。

## Phase 5：PVP 对位助手

### 目标

在对战中根据我方宠物、技能属性、伤害表现，辅助判断对方宠物可能属性或属性倍率。

### 要做的功能

- 选择我方宠物和技能。
- 输入或选择对位的敌方宠物候选。
- 记录伤害表现或倍率线索，例如明显克制、抵抗、中性、异常偏高/偏低。
- 输出“可能性分析”，例如可能弱某属性、可能抵抗某属性、可能存在属性倍率影响。
- 为后续更复杂的伤害计算保留接口。

### 暂不做的内容

- 不承诺精准反推对方属性。
- 不做完整实战公式模拟。
- 不覆盖天气、护盾、随机浮动、连击特判等复杂因素。
- 不做自动识别战斗画面。

### 验收标准

- 能基于技能属性和观测结果给出可能性分析。
- 输出文案明确是辅助判断，不是确定结论。
- 分析结果能链接回属性克制、图鉴或技能信息。
- PVP 模块在手机、iPad、桌面可用。

### 对上游同步风险的评估

风险低到中等。建议作为 `src/features/pvp` 下的独立模块开发，只复用基础数据读取和适配层，不直接改 `public/data`、`scripts` 或核心数据生成逻辑。

## 明确延后或不做的方向

- 每日任务、每周任务、打卡提醒：不作为个人版核心方向，除非未来需求明确变化。
- 收藏、备注：当前暂不需要，后续如果需要再放入 `src/custom`、`public/my-data` 或 `localStorage`。
- 大规模手工维护基础 JSON：不做。基础数据仍跟随上游和脚本生成流程。

## 下一步建议

1. 先设计 Phase 1 的首页信息架构草图。
2. 新增 `src/features/my-home` 承载个人首页组件，减少直接改原页面的范围。
3. 同时规划 `src/features/battle-query` 的属性、图鉴、技能统一查询入口。
4. 在开始 PVP 前，先定义队伍 JSON 结构草案，避免后续反复迁移个人数据。
