# 个人版设计规范

本文记录个人版 `rocom.aoe` 的 UI 风格、页面组织和后续开发约束。目标是在继续扩展 PVP 等功能前，保持属性查询、图鉴、技能查询和未来对位辅助的体验一致。

## 整体定位

个人版定位为“洛克王国世界战斗资料助手”，优先服务战斗前和战斗中的快速查询。

- 核心能力优先级：属性查询、图鉴、技能查询、未来 PVP 对位辅助。
- 页面设计优先清晰、稳定、可扫描，不追求复杂动效或装饰性视觉。
- 保留原项目工具入口，但首页和主导航应突出个人高频使用路径。
- 不把每日任务、打卡、强提醒或让人有负担的功能作为个人版核心方向。

## 页面风格

- 使用现有暗色背景、卡片式布局、边框和阴影体系。
- 信息密度保持适中：首屏突出入口和关键判断，详情信息放在后续区域。
- 视觉层级应服务查询效率：标题清楚、说明短、标签可扫、操作入口明确。
- 避免大面积重做上游页面 UI，优先通过新增 `src/features/*` 组件轻量接入。
- 不为了视觉效果引入新依赖，不新增纯装饰性的复杂背景。

## 首页入口规范

首页第一优先级固定为四个核心入口：

- 属性查询：进入 `/attributes`，用于属性克制、单属性四面、双属性打击面和防守面。
- 图鉴：进入 `/encyclopedia`，用于宠物资料、属性、血脉技能和详情页入口。
- 技能查询：进入 `/skills`，用于技能名称、ID、描述、属性和分类查询。
- PVP 工具：保持 Coming Soon，不提前实现复杂 PVP 功能。

“更多工具”区域继续承载原项目能力，例如配队、配种、孵蛋/查蛋、星图、宠物表格、道具和图鉴进度。新增个人功能时，只有高频战斗查询才进入首页核心入口。

## 卡片规范

卡片结构保持一致，优先包含这些层级：

- 标题：用短词说明卡片对象，例如“属性查询”“技能查询”。
- 说明：用一到两句话说明能解决的问题，不写过长教程。
- 标签：用于呈现状态、分类、属性、倍率、来源等可快速扫描的信息。
- 操作入口：用明确按钮、链接或整卡点击进入下一步。

视觉约束：

- 圆角、边框、背景和阴影复用现有 `Card`、`Badge`、`Button` 等组件风格。
- 同组卡片高度、间距和信息层级尽量统一。
- 卡片不要嵌套卡片；需要分组时使用标题、分隔线或网格布局。
- 结果卡片优先突出核心判断信息，再展示补充字段。

## 标签规范

标签用于压缩信息和辅助扫读，不承载长句。

- 属性标签：展示属性名称、属性倍率、克制/被克制关系等战斗判断。
- 分类标签：展示技能分类、来源、筛选分组或宠物状态。
- 状态标签：展示“高频查询”“资料核心”“技能入口”等入口状态。
- Coming Soon 标签：仅用于规划中但尚未实现的入口，不能伪装成可用功能。

标签文本应短，颜色语义应稳定。同一语义在不同页面尽量使用相同视觉倾向，例如风险、优势、中性、规划中不要混用。

## 查询页规范

查询页默认从输入和筛选开始，再展示结果。

- 搜索框放在前面，支持名称、ID 和关键描述等高频关键词。
- 筛选项紧跟搜索框，避免用户先看到大量结果再找筛选。
- 结果卡片要突出核心判断信息，例如属性倍率、技能属性、技能分类、威力和关键描述。
- 空状态要说明当前条件没有结果，并提示放宽关键词或筛选。
- 加载状态要清楚，避免页面看起来像空白或数据丢失。
- 错误状态要给出可理解的原因，不把底层异常直接暴露给普通使用路径。

## 移动端规范

- 手机、iPad 和桌面都不能横向溢出。
- 卡片、筛选区和结果列表允许纵向堆叠，优先保证阅读和点击空间。
- 筛选区不要过度拥挤；必要时把高级筛选折叠或分组。
- 按钮、输入框和标签需要留足触控空间，不依赖精细 hover 操作。
- 长文本必须换行或截断，不遮挡后续内容。
- PWA 主屏幕模式下不要依赖浏览器地址栏或后退栏提示，页面内导航和状态反馈必须自洽。

## PWA 规范

- PWA 优先服务离线查询，不做 Capacitor 或 iOS 原生壳。
- 首页、属性查询、图鉴和技能查询应保持基础离线可打开。
- 核心 JSON 可以预缓存；大量宠物详情、图片和图标只做有限 runtime cache。
- 离线状态下数据未缓存时，页面应显示清楚的加载失败或降级提示，不应白屏。
- 更新策略优先保证在线时能刷新缓存，避免用户长期卡在旧数据上。

## PVP 页面规范

后续 PVP 页面先做辅助判断，不做复杂大而全。

- 优先围绕对位、属性倍率、技能属性和观察记录提供辅助判断。
- 先明确数据结构和边界，再实现导入、导出或计算逻辑。
- 不做每日任务、每周任务、打卡、强提醒。
- 不做让人有负担的功能，例如必须维护大量手动状态才能使用的流程。
- 不把 PVP 伤害公式、队伍导入导出和技能查询混在同一个首版页面里。
- PVP 相关个人数据不要写入 `public/data`，应放入 `src/features/pvp`、`src/custom` 或 `public/my-data` 等隔离区域。

## 开发约束

- 使用 Yarn，不运行 `npm install`。
- 优先复用现有组件、布局、工具函数和 `src/features/*` 结构。
- 不大改上游核心页面结构；确实需要扩展时，优先新增组件再轻量接入。
- 不为了视觉效果引入新依赖。
- 不手工修改 `public/data` 中的基础游戏数据。
- 不随意修改 `scripts`，除非任务明确涉及数据同步或脚本维护。
- 不修改 `package.json` 或 `yarn.lock`，除非任务明确涉及依赖。
- 每次功能改动同步更新 `docs/CHANGELOG.md`；文档类改动也应记录。
